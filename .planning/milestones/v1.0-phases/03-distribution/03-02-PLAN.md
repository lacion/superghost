<task type="auto" tdd="true">
  <name>Task 1: Cross-platform build script and binary compilation integration test</name>
  <files>scripts/build-binaries.sh, tests/integration/binary-build.test.ts</files>
  <behavior>
    - Build script creates dist/ directory
    - Build script runs bun build --compile for all 4 targets with --minify --bytecode --sourcemap flags
    - Build script names output files as superghost-{os}-{arch} (e.g., superghost-darwin-arm64)
    - Integration test: bun build --compile succeeds for the host platform (exit code 0)
    - Integration test: compiled binary file exists and is executable
    - Integration test: compiled binary runs --version without error (exit code 0)
  </behavior>
  <action>
    1. Create `scripts/build-binaries.sh`:
       ```bash
       #!/usr/bin/env bash
       set -euo pipefail

       # Build standalone binaries for all target platforms
       # Uses Bun's cross-compilation: all builds run on a single machine

       ENTRY="src/cli.ts"
       OUTDIR="dist"
       TARGETS=(
         "bun-darwin-arm64:superghost-darwin-arm64"
         "bun-darwin-x64:superghost-darwin-x64"
         "bun-linux-arm64:superghost-linux-arm64"
         "bun-linux-x64:superghost-linux-x64"
       )

       mkdir -p "$OUTDIR"

       for entry in "${TARGETS[@]}"; do
         target="${entry%%:*}"
         outfile="${entry##*:}"
         echo "Building $outfile (target: $target)..."
         bun build --compile \
           --target="$target" \
           --minify \
           --bytecode \
           --sourcemap \
           "$ENTRY" \
           --outfile "$OUTDIR/$outfile"
       done

       echo "All binaries built in $OUTDIR/"
       ls -lh "$OUTDIR"/superghost-*
       ```
       - Make executable: `chmod +x scripts/build-binaries.sh`
       - Per locked decisions: target macOS (arm64 + x64) and Linux (arm64 + x64), Windows deferred
       - Per locked decision: binary naming `superghost-{os}-{arch}`
       - Per research: use `--minify --bytecode --sourcemap` flags for optimization

    2. Write integration test in `tests/integration/binary-build.test.ts`:
       - Test: "compiles standalone binary for host platform"
         a. Determine host target: construct from `process.platform` and `process.arch` (e.g., "bun-darwin-arm64")
         b. Determine output name: `superghost-${process.platform === "darwin" ? "darwin" : "linux"}-${process.arch === "arm64" ? "arm64" : "x64"}`
         c. Run `bun build --compile --target=${hostTarget} src/cli.ts --outfile dist/test-binary` via `Bun.spawn`
         d. Assert exit code is 0
         e. Assert output file exists: `Bun.file("dist/test-binary").exists()` is true

       - Test: "compiled binary runs --version"
         a. Depends on previous test (use `beforeAll` to build once)
         b. Run `./dist/test-binary --version` via `Bun.spawn`
         c. Assert exit code is 0
         d. Assert stdout contains "0.1.0" (version from package.json)

       - Use `afterAll` to clean up: delete `dist/test-binary`

       NOTE: This test takes ~10-15 seconds due to compilation. Use `test.timeout(30_000)` to account for compilation time. This is acceptable for an integration test that only runs in CI or manually.

    IMPORTANT per research pitfall 5: target strings like "bun-darwin-arm64" are easy to mistype. The build script uses an array of exact strings to avoid this.
    IMPORTANT per research pitfall 4: version synchronization -- the compiled binary should read version from package.json or use --define. The integration test verifies version output matches package.json.

<task type="auto">
  <name>Task 2: GitHub Actions release workflow for npm publish and binary distribution</name>
  <files>.github/workflows/release.yml</files>
  <action>
    Create `.github/workflows/release.yml` following research Pattern 4. This is a single workflow triggered by version tag push that handles both npm publishing and binary distribution.

    Structure:
    ```yaml
    name: Release

    on:
      push:
        tags:
          - "v*.*.*"

    permissions:
      contents: write
      id-token: write
    ```

    Jobs (3 jobs, `build-and-release` and `publish-npm` both depend on `test`):

    **Job 1: `test`**
    - runs-on: ubuntu-latest
    - Steps:
      a. `actions/checkout@v4`
      b. `oven-sh/setup-bun@v2`
      c. `bun install`
      d. `bun test`
      e. `bunx tsc --noEmit`

    **Job 2: `build-and-release`**
    - needs: test
    - runs-on: ubuntu-latest
    - permissions: contents: write
    - Steps:
      a. `actions/checkout@v4`
      b. `oven-sh/setup-bun@v2`
      c. Build all 4 platform binaries inline (same commands as build-binaries.sh but inlined for CI clarity):
         ```
         mkdir -p dist
         for target in bun-darwin-arm64 bun-darwin-x64 bun-linux-arm64 bun-linux-x64; do
           os_arch="${target#bun-}"
           bun build --compile \
             --target="$target" \
             --minify \
             --bytecode \
             --sourcemap \
             src/cli.ts \
             --outfile "dist/superghost-$os_arch"
         done
         ```
      d. Create GitHub Release + upload binaries:
         ```
         uses: softprops/action-gh-release@v2
         with:
           generate_release_notes: true
           files: dist/superghost-*
         ```

    **Job 3: `publish-npm`**
    - needs: test
    - runs-on: ubuntu-latest
    - permissions: contents: read, id-token: write
    - Steps:
      a. `actions/checkout@v4`
      b. `oven-sh/setup-bun@v2`
      c. `actions/setup-node@v4` with node-version: "24", registry-url: "https://registry.npmjs.org"
      d. `bun install`
      e. `npm publish --provenance --access public`

    Key design decisions (Claude's discretion per CONTEXT.md):
    - Release tag format: `v*.*.*` (e.g., v0.1.0) — standard semver with v prefix
    - Jobs 2 and 3 run in parallel (both depend only on `test`) for faster releases
    - Build commands inlined in workflow rather than calling build-binaries.sh — makes CI self-documenting and avoids script permission issues
    - Uses `generate_release_notes: true` for automatic release notes from commit history
    - Node.js 24 for npm >= 11.5.1 (required for OIDC trusted publishing per research pitfall 3)

    IMPORTANT per locked decision: Same release tag triggers BOTH binary builds AND npm publish
    IMPORTANT per locked decision: Binaries distributed via GitHub Releases only (no Homebrew, no install script)
    IMPORTANT per research: npm OIDC trusted publishing eliminates long-lived NPM_TOKEN; requires npm >=11.5.1 and id-token: write permission
    IMPORTANT per research pitfall 6: softprops/action-gh-release creates release AND uploads in one step — no separate release creation needed

    NOTE: The user must configure npm trusted publishing on npmjs.com before the first publish. This is a one-time setup linking the GitHub repo to the npm package. The workflow will fail without it — but the error message from npm is clear.
  </action>
