  <name>Task 1: McpManager for Playwright and curl MCP server lifecycle</name>
  <files>src/agent/mcp-manager.ts</files>
  <action>
    Create `src/agent/mcp-manager.ts` following the pattern from 02-RESEARCH.md Pattern 1.

    1. Import `createMCPClient` from "@ai-sdk/mcp"
    2. Import `StdioClientTransport` from "@modelcontextprotocol/sdk/client/stdio.js" — NOT `Experimental_StdioMCPTransport` from "@ai-sdk/mcp" (Node.js only, will not work with Bun per research pitfall 2)
    3. Import Config type from "../config/types.ts"

    4. Export class `McpManager`:
       - Private fields: `playwrightClient` and `curlClient` (both typed as `Awaited<ReturnType<typeof createMCPClient>> | null`, initialized to null)
       - Private field: `config` of type `Pick<Config, "browser" | "headless">`
       - Constructor accepts config (Pick<Config, "browser" | "headless">)

       - Method `async initialize(): Promise<void>`:
         a. Spawn Playwright MCP via createMCPClient with StdioClientTransport:
            - command: "bunx"
            - args: ["@playwright/mcp@latest", "--headless" (only if config.headless), "--isolated", `--browser=${config.browser}`]
            - Only include "--headless" flag if config.headless is true
         b. Spawn curl MCP via createMCPClient with StdioClientTransport:
            - command: "bunx"
            - args: ["@calibress/curl-mcp"]
         c. Assign both to class fields

       - Method `async getTools(): Promise<Record<string, any>>`:
         a. Get tools from both clients: `await this.playwrightClient!.tools()` and `await this.curlClient!.tools()`
         b. Merge and return: `{ ...playwrightTools, ...curlTools }`
         c. This provides ALL tools to the agent regardless of test type (per CONTEXT.md: "Both Playwright MCP and curl MCP servers launched for every test regardless of type")

       - Method `async close(): Promise<void>`:
         a. `await Promise.allSettled([this.playwrightClient?.close(), this.curlClient?.close()])`
         b. Set both to null

    IMPORTANT: MCP servers are shared across the test suite (not restarted per test). Fresh browser context per test comes from the `--isolated` flag on Playwright MCP. Per research open question 2, if shared servers cannot provide per-test isolation, we may need to revisit, but `--isolated` is the starting approach.

    NOTE: This component is NOT unit tested with mocks because MCP server spawning is inherently an integration concern. The agent-runner tests will mock McpManager's getTools() return value. Full integration testing of MCP lifecycle happens in Phase 2 verification.
  </action>

  </verify>
  <done>McpManager compiles, spawns Playwright and curl MCP servers via Bun-compatible StdioClientTransport, provides merged tool set, handles cleanup via Promise.allSettled.</done>

       - Mock `generateText` from "ai" module to return controlled responses
       - Test: when generateText returns output { passed: true, message: "Login works" }, executeAgent returns AgentExecutionResult with passed=true, message, and steps from recorder
       - Test: when generateText returns null output, executeAgent returns passed=false with "did not produce a structured result" message
       - Test: executeAgent passes recursionLimit to stopWhen
       - Test: executeAgent wraps tools with recorder (verify recorder.wrapTools was called)
       - Test: steps from recorder are included in the returned AgentExecutionResult
       - NOTE: These tests mock generateText at the module level. The actual AI call is an integration concern.
       - Use `import { mock } from "bun:test"` for mocking

    2. Implement `src/agent/agent-runner.ts` (GREEN):
       - Import `generateText`, `Output`, `stepCountIs` from "ai"
       - Import `z` from "zod"
       - Import `StepRecorder` from "../cache/step-recorder.ts"
       - Import `AgentExecutionResult` from "./types.ts"
       - Import `buildSystemPrompt` from "./prompt.ts"

       - Define `TestResultSchema = z.object({ passed: z.boolean(), message: z.string() })`

       - Export `async function executeAgent(config: { model: any; tools: Record<string, any>; testCase: string; baseUrl: string; recursionLimit: number; globalContext?: string; testContext?: string }): Promise<AgentExecutionResult>`:
         a. Create new StepRecorder
         b. Wrap tools: `const wrappedTools = recorder.wrapTools(config.tools)`
         c. Build system prompt: `buildSystemPrompt(config.testCase, config.baseUrl, config.globalContext, config.testContext)`
         d. Call generateText:
            ```
            const { output, steps } = await generateText({
              model: config.model,
              tools: wrappedTools,
              system: systemPrompt,
              prompt: `Execute the test case: "${config.testCase}"`,
              stopWhen: stepCountIs(config.recursionLimit),
              output: Output.object({ schema: TestResultSchema }),
            });
            ```
         e. If output is null (agent exhausted steps without structured output):
            Return `{ passed: false, message: "Agent did not produce a structured result — it may have exceeded the ${config.recursionLimit} step limit", steps: recorder.getSteps() }`
         f. Return `{ passed: output.passed, message: output.message, steps: recorder.getSteps() }`

    IMPORTANT per CONTEXT.md: Structured JSON output via Vercel AI SDK. No text marker parsing.
    IMPORTANT per research: Output.object() generation counts as one step in the stopWhen budget. The recursionLimit value from config accounts for this.
    IMPORTANT: Use `generateText` with `Output.object()`, NOT deprecated `generateObject`.
    IMPORTANT: Use `stopWhen: stepCountIs(n)`, NOT deprecated `maxSteps`.
