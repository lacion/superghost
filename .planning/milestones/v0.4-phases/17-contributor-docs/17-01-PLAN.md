<task type="auto">
  <name>Task 1: Create CONTRIBUTING.md and SECURITY.md</name>
  <files>CONTRIBUTING.md, SECURITY.md</files>
  <action>
Create CONTRIBUTING.md at repo root with the following sections (friendly and direct tone, similar to Bun/Vite contributing guides):

1. **Welcome** -- brief welcome, link to issues for questions
2. **Prerequisites** -- Bun installation instructions (link to bun.sh), Git
3. **Dev Setup** -- step-by-step: clone, `bun install`, `bun test` (show expected output like "X tests passed"), `bunx biome check .`, `bunx tsc --noEmit`
4. **Project Architecture** -- brief directory map showing:
   ```
   src/
     cli.ts          -- CLI entry point (Commander.js)
     agent/          -- AI agent execution (LLM tool calls)
     cache/          -- Step-level caching (.superghost-cache/)
     config/         -- YAML config loading, env var interpolation
     dist/           -- Binary build scripts
     infra/          -- Infrastructure utilities
     output/         -- Output formatters (json, junit, reporter)
     runner/         -- Test runner orchestration
   ```
5. **Adding a New Output Format** -- mini tutorial pointing to `src/output/json-formatter.ts` as the simplest complete example, explaining the batch-formatter pattern (formatter receives all results, writes to stdout, progress goes to stderr)
6. **Linting and Formatting** -- `bunx biome check .` to check, `bun run lint:fix` to auto-fix, Biome as the single lint/format/import-sort tool
7. **Testing** -- `bun test` for unit tests, note about `--frozen-lockfile` in CI
8. **Pull Request Process** -- fork, branch, make changes, ensure `bun test` + `bunx biome check .` + `bunx tsc --noEmit` pass, open PR with meaningful description. Note that CI runs these checks automatically.

ALL commands MUST use `bun`/`bunx` (never npm/npx) per project constraint.

Create SECURITY.md at repo root with:

1. **Reporting a Vulnerability** -- use GitHub private Security Advisory at `https://github.com/lacion/superghost/security/advisories/new` (no email)
2. **Response Timeline** -- "We'll respond as soon as we can" (best-effort, honest for solo-maintainer)
3. **Scope** -- what counts as security (secret leakage in cache, cache poisoning, dependency vulnerabilities) vs regular bugs (crash bugs, config parse errors)
4. **Supported Versions** -- latest release only, no backporting

Create `.github/ISSUE_TEMPLATE/` directory (must use `.yml` extension, NOT `.yaml` -- GitHub requires this).

Create `.github/ISSUE_TEMPLATE/bug_report.yml` as a YAML form template with:
- `name: Bug Report`
- `description: Report a bug in SuperGhost`
- `labels: ["bug"]`
- Body fields (each with unique `id`):
  - `markdown`: brief thanks/instructions
  - `textarea` id=description: "Description" (required)
  - `textarea` id=steps: "Steps to Reproduce" with placeholder showing example commands (required)
  - `textarea` id=expected: "Expected Behavior" (required)
  - `textarea` id=actual: "Actual Behavior" (required)
  - `textarea` id=config: "YAML Configuration" with `render: yaml` for code display, description "Paste your SuperGhost YAML config (remove any secrets)" (not required)
  - `input` id=superghost-version: "SuperGhost Version" placeholder "0.3.1" (required)
  - `input` id=bun-version: "Bun Version" placeholder "1.3.x" (required)
  - `input` id=os: "Operating System" placeholder "macOS 15.2, Ubuntu 24.04, etc." (required)

Create `.github/ISSUE_TEMPLATE/feature_request.yml` as a YAML form template with:
- `name: Feature Request`
- `description: Suggest a feature or improvement`
- `labels: ["enhancement"]`
- Body fields:
  - `textarea` id=problem: "What problem does this solve?" with description "Describe the problem or pain point" (required)
  - `textarea` id=solution: "Describe your ideal solution" (required)
  - `textarea` id=alternatives: "Alternatives considered" with description "Any other approaches you've thought about" (not required)

Create `.github/ISSUE_TEMPLATE/config.yml` with:
- `blank_issues_enabled: true`

Create `.github/pull_request_template.md` with:
- `## Description` section with HTML comment prompt
- `## Related Issue` section with "Related issue: #" (noted as optional)
- `## Type of Change` checkboxes: Bug fix, New feature, Breaking change, Documentation
- `## Checklist` with three items: tests pass (`bun test`), lint passes (`bunx biome check .`), PR has a meaningful description
- `## How to Test` section with HTML comment prompt for manual verification steps

Use only `bug` and `enhancement` labels (GitHub defaults, should exist on repo).
